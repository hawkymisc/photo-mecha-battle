# 06. 課金・公平性仕様

[← 仕様書一覧](00_root_overview.md)

## 目的

本ドキュメントは、課金要素、Pay to Win 回避方針、公平性に関する仕様を定義する。

> **確定状態（2026-07-02 更新）**: 生成クォータの数値と premium 拡大を公式仕様に確定した（PO 判断）。
> Entitlement は本書の設計を正とし、現行実装（スタブ）との対応を「実装状態」列に明記する。

## 基本方針

本作の課金は、戦闘能力の直接強化ではなく、利便性、表現、保存、生成支援に寄せる。

課金で強くなるのではなく、作戦作成や整理が楽になる設計とする。

## 課金で提供してよいもの

| 課金要素 | 内容 |
|---|---|
| 自然言語戦術生成 | 文章から戦術スロットを生成 |
| 戦術名の自動提案 | 作戦名を生成 |
| 戦術説明の自動生成 | 作戦内容を説明文にする |
| 戦術保存枠追加 | 複数の戦術を保存しやすくする |
| ログ要約 | バトル結果を読みやすく要約 |
| 見た目カスタム | AIコア、UIスキン、カード装飾 |
| 生成画像の再生成枠 | 見た目の再生成。ただし要上限 |
| 生成クォータ拡大 | 撮影・メカ生成の日次上限を引き上げる（下記「生成クォータ」） |

## 課金で提供しないもの

| 項目 | 理由 |
|---|---|
| 戦闘ステータス強化 | Pay to Win になる |
| 使用可能条件の追加 | 戦術バリエーションが課金で増える |
| 使用可能行動の追加 | 戦術バリエーションが課金で増える |
| ランク戦スロット数増加 | 課金者が複雑な AI を使えるようになる |
| 戦闘中の判断精度向上 | 課金者だけ強い AI になる |
| 非公開情報の閲覧 | 競技性を壊す |

## Pay to Win 回避原則

以下を明文化する。

```text
課金しても、ランク戦における戦術スロット数、使用可能条件、使用可能行動、バトル中の判断能力は変化しない。
```

## 生成クォータ（確定）

撮影アップロードとメカ生成には日次クォータを設ける（生成コスト・不正対策。[`docs/08`](08_mvp_and_roadmap.md)）。
消費は UTC 日付単位でカウントし、翌日リセットする。超過時は API が 429 を返す。

| 枠 | 無課金 | premium |
|---|---:|---:|
| 撮影アップロード / 日 | 20 | 50 |
| メカ生成 / 日 | 10 | 30 |

**公平性の整理**: クォータ拡大は「生成の試行回数」を増やすのみである。
1 体あたりのステータス算出式（[`docs/03`](03_mech_generation_and_stats.md)）、ランク戦チームの 3 体編成、戦術スロット数・条件・行動
（[`docs/04`](04_tactics.md)）は全プレイヤー共通であり、戦闘中の判断能力に差はつかない。
よってクォータ拡大は Pay to Convenience の範囲内とする。
ただし Paywall では「バトル性能は変わらない」ことを明示する（下記 Paywall 方針）。

## 自然言語戦術生成の位置づけ

自然言語戦術生成は、戦術作成の入力補助である。

課金ユーザーが文章で作った戦術も、最終的には通常の戦術スロット（[`docs/04`](04_tactics.md) の DSL）へ変換される。
非課金ユーザーも手動で同じ戦術を作れる。

## ローカル LLM とクラウド推論

### ローカル LLM 採用時

- 課金ユーザーは回数制限なしで利用可能
- サーバー推論コストが増えない
- プライバシー面で有利
- 端末差、発熱、容量が課題

### クラウド推論採用時

- 品質を安定させやすい
- 古い端末でも利用しやすい
- サーバーコストが発生する
- 回数制限を設ける必要がある

## 判断基準

ローカル LLM の採用可否は以下で評価する。

| 項目 | 判断基準 |
|---|---|
| 変換精度 | 4 スロット + 基本行動への変換成功率が十分か |
| 応答速度 | スマートフォン上で許容時間内に返るか |
| 端末差 | 古い端末で実用可能か |
| バッテリー | 発熱・消費電力が許容範囲か |
| アプリ容量 | モデル同梱による容量増が許容範囲か |
| 安定性 | iOS/Android で挙動差が小さいか |

## プレイヤーへの説明

ゲーム内では以下のように説明する。

```text
自然言語作成は、作戦スロットの入力補助機能です。
バトル中の判断能力や使用可能な行動は変わりません。
```

この説明により、課金機能が高性能 AI ではなく利便性機能であることを明確にする。

## RevenueCat 必須仕様

ハッカソン要件により、本作の課金基盤には RevenueCat を必ず組み込む。

### RevenueCat の役割

| 領域 | 内容 |
|---|---|
| アプリ内課金 | iOS / Android の購入処理を統合管理 |
| サブスクリプション | 月額・年額プランの管理 |
| Entitlement | 課金機能の利用権限管理 |
| 購入復元 | 端末変更・再インストール時の権限復元 |
| Offerings | 表示する課金プランの管理 |
| Webhook | バックエンドへの課金状態通知 |
| 分析 | 課金イベントの確認、ハッカソン審査用デモ補助 |

### Entitlement 定義（正）と実装状態

| Entitlement | 解放機能 | 実装状態 |
|---|---|---|
| `premium_tactics` | 自然言語戦術生成（Phase 3） | キーは実装済み。機能ゲート対象（自然言語生成）が未実装 |
| `generation_boost` | 生成クォータ拡大（上記の表） | **実装済み**（[`api/limits.py`](../src/photo_mecha_battle/api/limits.py)） |
| `extra_tactic_slots` | 戦術**保存枠**の拡張 | 未実装（Phase 3） |
| `battle_log_summary` | バトルログ要約 | 未実装（Phase 3） |
| `cosmetic_pack_access` | UI スキン、カード装飾などの見た目要素 | 未実装（Phase 3+） |

注意：`extra_tactic_slots` は「保存枠」の拡張であり、ランク戦で使用できる戦術スロット数を増やすものではない。

> **現行実装の注記**: 現行の webhook スタブは購入系イベントで `premium_tactics` と `generation_boost` を
> 一括で有効化する簡易実装である。また、クォータ拡大の判定も `premium_tactics` **または** `generation_boost`
> のいずれかが有効なら適用される（[`api/limits.py`](../src/photo_mecha_battle/api/limits.py)）。本仕様では**クォータ拡大は `generation_boost` のみ**に
> 紐づくのが正であり、商品と Entitlement の対応が RevenueCat ダッシュボードで確定した時点で、
> イベント内の entitlement 情報に基づく個別付与＋キー別の機能ゲートに置き換える（PLAN D-005 参照）。

### Webhook イベント処理（MVP）

| RevenueCat イベント | サーバー処理 |
|---|---|
| `INITIAL_PURCHASE` / `RENEWAL` / `NON_RENEWING_PURCHASE` / `PRODUCT_CHANGE` | 対象 Entitlement を有効化 |
| `CANCELLATION` / `EXPIRATION` | 対象 Entitlement を無効化 |
| 上記以外 | 無視（`applied: false` を返す） |
| `app_user_id` がサーバーの user と一致しない | `applied: false, reason: user_not_found` |

### Product / Package 案

| 商品 | 内容 |
|---|---|
| Monthly Premium | 自然言語戦術生成、保存枠拡張、ログ要約、生成クォータ拡大 |
| Annual Premium | Monthly Premium の年額版 |
| Cosmetic Pack | カード装飾・AI コア外観 |
| One-time Support Pack | ハッカソン向け買い切り支援パック。性能差は持たせない |

### Paywall 方針

Paywall では、以下を明示する。

```text
Premium は作戦作成とメカ生成を便利にする機能です。
バトル中の判断能力、使用可能条件、使用可能行動、ランク戦スロット数は変わりません。
```

### 実装上の必須要件

- iOS / Android 双方に RevenueCat SDK を組み込む。
- App Store Connect / Google Play Console の商品定義と RevenueCat Product を対応させる。
- RevenueCat Offerings を使用して、アプリ側の表示プランを管理する。
- アプリ起動時またはログイン時に CustomerInfo を取得し、Entitlement 状態を確認する。
- 購入復元ボタンを必ず用意する。
- バックエンドは RevenueCat Webhook を受け取り、ユーザーの課金状態を同期する。
- 課金状態はクライアントのみを信用せず、サーバー側でも検証・保持する。

### デモ用 Entitlement 付与 API の扱い

ハッカソンデモ用に、Entitlement を直接切り替える API（`POST /billing/entitlements`）を用意している
（[`docs/08`](08_mvp_and_roadmap.md) の「Entitlement 強制付与できる管理者フラグ」に対応）。

- **現状は認証ユーザーが自分自身の Entitlement を自由に変更できる**。デモ・開発環境限定の挙動である。
- 本番公開前に、管理者権限チェックまたはエンドポイント無効化を必須とする（PLAN の修正タスク参照）。
