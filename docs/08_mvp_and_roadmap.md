# 08. MVP・ロードマップ・リスク仕様

[← 仕様書一覧](00_root_overview.md)

## 目的

本ドキュメントは、MVP範囲、開発フェーズ、技術リスク、未決事項を定義する。

## MVPで実装するもの

- iOS/Android向け基本アプリ
- 写真撮影
- 写真アップロード
- 1オブジェクト選択
- セグメンテーション
- 情報量・特徴量計算
- 特徴量からのメカ型自動推定（[`docs/03`](03_mech_generation_and_stats.md) 型推定ルール）
- メカ画像生成
- メカ保存
- 初期戦術プリセット5種
- 4スロット + 基本行動の戦術編集
- 3体チーム編成
- 非同期オートバトル
- バトルログ表示
- 簡易ランキング

## MVPでは実装しないもの

- リアルタイムPvP
- 複数オブジェクト合成
- トレード
- マーケット
- 位置情報連動
- AR表示
- 本格的な育成ツリー
- 複雑なノーコードAIエディタ
- 戦闘中LLM判断
- 高度なギルド機能

## MVP後に検討するもの

- 自然言語戦術生成
- ローカルLLM検証
- 戦術共有コード
- イベントリーグ
- 3対3以外のルール
- 支援型・妨害型などの追加プリセット
- カード装飾
- ログ要約
- 作戦改善提案

## 開発フェーズ

### Phase 0：技術検証

目的：主要技術の実現性確認。

検証内容：

- 写真からのオブジェクト抽出
- セグメンテーション品質
- 情報量スコアの妥当性
- i2iメカ生成の安定性（調査・検証計画: [`docs/10`](10_mobile_image_generation_survey.md)）
- ローカルLLMによる戦術スロット変換
- iOS/Android端末上での処理性能（同上）

### Phase 1：縦切りプロトタイプ

目的：ゲームループの最小成立確認。

実装内容：

- 撮影
- オブジェクト選択
- 特徴量分析・型自動推定
- メカ生成
- 戦術プリセット選択
- ダミーバトル
- バトルログ表示

### Phase 2：MVP

目的：限定ユーザーで遊べる状態にする。

実装内容：

- ユーザー登録
- メカ保存
- 戦術編集
- チーム編成
- 非同期PvP
- ランキング
- 初期不正対策
- 生成回数制限

### Phase 3：β版

目的：継続性と課金導線の検証。

実装内容：

- 自然言語戦術生成
- ローカルLLMまたはクラウドLLM
- 戦術保存枠
- 戦術共有
- 戦闘ログ要約
- イベントバトル

### Phase 4：正式版

目的：運用・拡張可能なタイトルとして展開。

実装内容：

- シーズン制
- イベントリーグ
- 追加戦術プリセット
- 外見カスタム
- 高度なランキング
- コミュニティ機能

## 技術リスク

### セグメンテーション品質

任意オブジェクトの抽出が不安定だと、ゲーム体験の根幹が崩れる。

対策：

- 自動検出と手動補正の併用
- マスク品質スコアの導入
- 抽出失敗時の再選択導線

### 情報量スコアの納得感

スコアがプレイヤーの感覚とズレると不満につながる。

対策：

- スコア理由を表示する
- 情報量だけでなく品質・形状・希少性を加味する
- 極端な画像にキャップを設ける

### 画像生成コスト

i2i生成はサーバーコストが高くなる可能性がある。

対策：

- 低解像度プレビュー
- 確定時のみ高解像度化
- 非同期ジョブ化
- 生成回数制限
- キャッシュ

### ローカルLLMの実用性

端末差、発熱、アプリ容量、変換品質が課題となる。

対策：

- 小型モデルで検証
- ルールDSLへの変換に特化
- クラウド推論とのハイブリッド案を用意
- 古い端末ではクラウドまたは機能制限を検討

### Pay to Win認識

課金が戦術作成を効率化するため、プレイヤーから有利と見なされる可能性がある。

対策：

- 使用可能条件・行動・スロット上限は完全共通
- 自然言語生成は入力補助であると明示
- 手動でも同じ戦術を作成可能にする
- 基本バトルログは全員に提供する

## 未決事項

> 以下の未決事項は [`config/po_pending_decisions.json`](../config/po_pending_decisions.json) に構造化データとして
> 一元管理する（PO 決定待ちの追跡用）。決定後は本ファイルと当該JSONの両方を更新する。
> RevenueCat のダッシュボード外部設定（商品定義・Webhook共有シークレット）は
> [`config/revenuecat_pending_setup.json`](../config/revenuecat_pending_setup.json) を参照。

### バトル関連

- 支援型・妨害型プリセットの導入時期
- 妨害（disrupt）の能力低下効果の仕様（状態異常システムと同時に設計）
- レーティングの Elo 式移行（現状は固定デルタ）
- 引き分け時の扱いの改善（現状はレーティング変動なし）

### 画像生成関連

- 採用する生成モデル（高品質 i2i 導入時）— 調査済み、PO 決定待ち（[`docs/10`](10_mobile_image_generation_survey.md)）
- 生成解像度
- 1体あたりの生成コスト
- 再生成回数
- 見た目とレアリティの関係（レアリティシステム自体が将来拡張）

### 戦術生成関連

- ローカルLLMの採用可否
- 対応端末範囲
- モデルサイズ
- 変換精度評価基準
- クラウド推論時の回数制限

### 課金関連

- 月額制か買い切りか
- 戦術保存枠の扱い（`extra_tactic_slots` の具体的な枠数）
- 生成画像再生成の課金有無
- ランク戦とカジュアル戦の課金差分
- 広告モデルの有無

## ハッカソン対応：RevenueCat 組み込み

ハッカソン要件として RevenueCat の組み込みを必須とする。

### MVPに追加する必須項目

- RevenueCat SDK の iOS / Android 組み込み
- RevenueCat Offerings 取得
- Paywall 表示
- サブスクリプションまたは買い切り商品の購入導線
- CustomerInfo による Entitlement 判定
- 購入復元
- RevenueCat Webhook 受信API
- サーバー側 Entitlement 状態保存
- 課金機能のデモ導線

### ハッカソン向け最小課金デモ

最小構成では、以下の1 Entitlementでよい。

| Entitlement | 内容 |
|---|---|
| `premium_tactics` | 自然言語戦術生成画面を解放する |

この場合でも、生成された戦術は通常の戦術スロットへ変換され、非課金ユーザーが手動で再現可能な範囲に限定する。

### Phase 0 に追加する検証

- RevenueCat SDK の iOS / Android 組み込み確認
- ストアのテスト環境でのテスト購入
- Google Play テスト購入確認
- 購入復元確認
- CustomerInfo 取得確認
- Entitlement による画面解放確認
- Webhook 受信確認

### リスク

| リスク | 対策 |
|---|---|
| ストア商品設定に時間がかかる | ハッカソン初日に App Store Connect / Google Play Console / RevenueCat を先に設定する |
| ストアのテスト環境でのテスト購入が不安定 | デモ用に Entitlement 強制付与できる管理者フラグを用意する |
| Webhook 構築が間に合わない | デモ付与は管理者フラグに限定。クライアント CustomerInfo の自己申告同期は採用しない（サーバー権威） |
| 課金がP2Wに見える | Paywallに「利便性機能であり戦闘性能は変わらない」と明記する |
