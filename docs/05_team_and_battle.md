# 05. チーム編成・バトル仕様

[← 仕様書一覧](00_root_overview.md)

## 目的

本ドキュメントは、チーム編成、位置、オートバトル、バトルログに関する仕様を定義する。

> **確定状態（2026-07-02 更新）**: バトル数値（ダメージ式、位置補正、行動順、ターン上限、EN 経済）は
> [`src/photo_mecha_battle/battle.py`](../src/photo_mecha_battle/battle.py) の実装値を正として確定した（バトル領域は実装還流方針）。
> 数値バランスの変更は本書とテスト（[`tests/test_battle*.py`](../tests/)）を同時に更新すること。

## チーム編成

基本編成は 3 体チームとする。

```text
チーム =
  前衛：機体A + 戦術セットA
  中衛：機体B + 戦術セットB
  後衛：機体C + 戦術セットC
```

- チームは前衛・中衛・後衛を各 1 体、計 3 体で必ず構成する（過不足は API が 400 で拒否）。
- 同一機体・同一戦術セットの重複使用の禁止は MVP では未実装（将来検討）。

## 位置

| 位置 | 狙われやすさ | 攻撃補正 | 防御補正 | 得意役割 |
|---|---|---:|---:|---|
| 前衛 | 高い（最優先で狙われる） | 1.00 | 1.10 | 近接、タンク、迎撃 |
| 中衛 | 普通 | 0.95 | 1.00 | 爆撃、妨害、バランス |
| 後衛 | 低い（前衛・中衛全滅後に狙われる） | 0.90 | 0.85 | 狙撃、支援、砲撃 |

- **攻撃補正**: 攻撃者の位置による与ダメージ倍率。前衛ほど高い。
- **防御補正**: 被攻撃者の位置による耐性。ダメージは `攻撃補正 ÷ 防御補正` で補正されるため、
  後衛は被弾時のダメージが大きい（0.85 で割る＝約 1.18 倍）代わりに、通常のターゲット選択では狙われにくい。

## ターゲット選択

- **主ターゲット**: 敵の生存機体を「前衛 → 中衛 → 後衛」の順で探索し、最初に見つかった機体。
- **後衛攻撃**: 敵後衛が生存していれば後衛を狙う。いなければ主ターゲット。
- **範囲攻撃**: 生存する敵全機体（最大 3 体）を対象とする。

## 弱点表示

編成結果に応じて、簡易的な弱点表示を行う（クライアント機能、MVP バックエンドの範囲外）。

```text

命中重視攻撃を持つ戦術を1つ入れると安定します。
```

基本助言は全プレイヤーに提供する（課金で差をつけない）。

## バトル基本方針

- バトルはオートで進行する。プレイヤーは戦闘中に直接操作しない。
- 戦闘結果はサーバー側で確定する（サーバー権威、[`docs/09`](09_lightweight_server_architecture.md)）。
- MVP では非同期 PvP を採用する。
- 乱数は 1 バトルにつき 1 つの疑似乱数生成器（`random.Random(seed)`）を使用し、
  同一 seed・同一入力でログまで完全に再現可能とする。乱数の消費順序も仕様の一部である。

## ターン構造

```text
1. ターン開始：全機体の防御・回避フラグを解除
2. 行動順を決定（後述）
3. 各機体が順に行動（戦術評価 → 行動解決 → ログ記録）
4. どちらかのチームが全滅したらバトル終了
5. 最大 30 ターンで決着しない場合は引き分け（勝者なし）
```

- 防御・回避の効果は**発動したターンの残りの被攻撃**にのみ有効
  （次ターン開始時に解除される。先に行動するほど長く守れる）。
- ターン途中で撃破された機体は、そのターンの残りの行動をスキップする。

## 行動順

同一ターン内の全生存機体（両チーム混合）を以下の優先で並べる。

1. SPD 降順（速い機体が先）
2. 同 SPD ならチーム ID の辞書順
3. さらに同順なら位置 ID（`back` → `front` → `middle`）の辞書順

※ 2・3 は決定性確保のためのタイブレークであり、ゲームデザイン上の意味は持たせない。
スキル・状態異常・型相性による行動順補正は MVP では実装しない（将来拡張）。

## EN（エネルギー）

| 項目 | 値 |
|---|---:|
| 初期値 | 最大 EN（`MechStats.en`）満タンで開戦 |
| 消費 | 行動ごとの EN コスト（[`docs/04`](04_tactics.md) の行動一覧参照） |
| チャージ | 1 回につき +25（最大 EN でキャップ） |
| 下限 | 0（負値にならない） |

EN が行動コストに満たない場合、その行動は選択できない（戦術評価で次のスロットへ）。

## ダメージ式（確定）

```text
BaseDamage = ATK × SkillPower × K / (DEF + 100)    # K = 60（DAMAGE_SCALING）

FinalDamage = int(max(1,
  BaseDamage
  × 行動特効補正        # 下記「行動特効」
  × 防御補正            # 対象が防御中なら 0.6
  × TypeModifier        # 型相性 1.15 / 1.00 / 0.90
  × PositionModifier    # 攻撃側攻撃補正 ÷ 対象側防御補正
  × RandomRange(0.9, 1.1)
  × CriticalModifier    # クリティカル時 1.2、それ以外 1.0
))
```

- **スケーリング係数 K**: `60`（[`battle.py`](../src/photo_mecha_battle/battle.py) の `DAMAGE_SCALING`）。
  現実的なステータス帯（ATK 58〜90 前後）で素点を実用域に乗せる係数。
  キャリブレーション例: ATK=80, DEF=50 → `80×60/150 = 32` ダメージ（下記表示例と整合）。
  K が無かった旧式では素点が常に 1.0 未満となり、最低ダメージ保証で全打撃が 1 に潰れて
  全バトルがターン上限の引き分けになっていた（PLAN D-010 / Issue #23）。
- **分母の `+100`**: ソフトアーマー正規化定数（「基本装甲 100」ではない）。DEF の限界効用を
  逓減させ、低 DEF 帯でのダメージ爆発を防ぐ。Issue #23 の調査結果に基づき維持。
- **式変更と過去バトルの互換性**: 過去バトルの正史は `battles.log_json` に保存済みの構造化ログ
  （D-003）であり、参照時に再シミュレーションしないため式変更の影響を受けない。
  同一 seed からの再現（リプレイ演出）が保証されるのは**同一の式・定数の範囲内のみ**。
  K などバランス定数を再調整する場合は、本節の注記とキャリブレーション例・テスト
  （[`tests/test_damage_scaling.py`](../tests/test_damage_scaling.py)）を同時更新すること。
- **最低ダメージ**: 命中した攻撃は必ず 1 以上のダメージを与える。
- **回避**: 対象が回避行動中の場合、35% の確率で被弾を完全回避（ダメージ 0）。
- **クリティカル率**: `LUCK / 500`（LUCK 100 で 20%）。倍率 1.2。
- **追撃（finisher）の減衰**: 対象 HP が 25% を超えている場合、SkillPower を 0.85 倍する
  （削り役への転用を防ぎ「倒し切り」用途に寄せる）。

### 行動特効

| 特効 | 条件 | 倍率 |
|---|---|---:|
| 命中重視攻撃 | 対象が鳥形 | 1.10 |
| 貫通攻撃 | 対象の DEF ≥ 80 | 1.15 |
| 迎撃 | 対象が前衛 | 1.10 |

## 型相性

```text
鳥形 > 獣型 > 人型 > 鳥形
```

| 状態 | 倍率 |
|---|---:|
| 有利 | 1.15 |
| 不利 | 0.90 |
| 同等 | 1.00 |

## 戦術評価

各機体はターン内の自分の手番開始時に戦術セットを評価する。

```text
1. 公開戦闘状態を取得（主ターゲットを決定）
2. スロット 1 から順に条件を評価
3. 条件を満たし、かつ EN コストを支払える最初の行動を選択
4. 支払えない場合は次のスロットを評価（フォールスルー）
5. すべて不成立なら基本行動（fallback）を実行
6. 行動を解決し、発動した条件と行動をログに記録
```

- 条件の一覧・意味・固定閾値は [`docs/04`](04_tactics.md) を正とする。
- 行動解決時点で EN が不足していた場合も基本行動へ退避し、ログの条件欄に「EN不足」と記録する。

## 戦術 AI が参照できる情報

| 情報 | 可否 |
|---|---|
| 自分の HP / EN / 位置 | 可 |
| 相手の型 | 可 |
| 相手の公開ステータス（DEF・SPD の高低判定含む） | 可 |
| 相手の現在 HP | 可 |
| 相手の位置 | 可 |
| 相手が防御中かどうか | 可 |
| 相手の非公開戦術（スロット内容） | 不可 |
| 乱数 seed | 不可 |
| 相手の次行動 | 不可 |

※「相手の直前行動」の参照は MVP では防御中フラグのみ（`target_defending`）。
行動履歴全般の参照条件は将来拡張。

## バトルログ

戦術改善のため、各ターンの行動理由を表示する。

### ログエントリの構造（正）

1 行動につき以下を記録する。

| フィールド | 内容 |
|---|---|
| turn | ターン番号（1 始まり） |
| actor_team | 行動チーム ID |
| actor_position | 行動機体の位置 |
| actor_name | 行動機体名 |
| condition_label | 成立した条件のラベル（例:「自分HPが30%以下」「基本行動」「EN不足」） |
| action | 実行した行動 ID |
| damage_events | 対象ごとの（対象 ID、対象名、ダメージ、撃破フラグ）の列 |
| note | 非ダメージ行動の効果説明（「防御態勢を取る」「ENを25回復」等） |

> **実装状態（PLAN D-003 対応済み）**: `battles.log_json` に上記フィールドを持つ配列を JSON 保存し、
> `GET /battles/{id}` のレスポンス `log_entries` として返す（[`battle_log_serde.py`](../src/photo_mecha_battle/battle_log_serde.py)）。
> 表示用の整形テキスト `log_text`（レスポンスの `log`）も引き続き保存する。
> マイグレーション前の既存行（`log_json` が NULL）は `log_entries: null` を返し、呼び出し側は `log` にフォールバックする。

### 表示例

```text
Turn 1
前衛：条件「相手が接近型」が成立
→ 迎撃を実行
→ 敵前衛に32ダメージ

Turn 2
中衛：条件「敵が複数残っている」が成立
→ 範囲攻撃を実行
→ 敵前衛に18ダメージ、敵中衛に14ダメージ

Turn 3
後衛：条件「自分ENが80以上」が成立
→ 狙撃を実行
→ 敵後衛に41ダメージ（撃破）
```

## ログの目的

- 戦術スロットが意図通り発動しているか確認する
- 発動しない条件を見つける
- 優先順位の問題を発見する（先勝ち評価では広い条件が狭い条件を隠す。[`docs/04`](04_tactics.md) の設計指針参照）
- チーム編成の弱点を理解する
- 次の戦術編集につなげる

## レーティング反映（MVP）

ランク戦（`POST /battles/ranked`）の結果はサーバーが確定し、以下の固定デルタで反映する。

| 結果 | 自分 | 相手 |
|---|---:|---:|
| 対人戦勝利 | +25 | −15 |
| 対人戦敗北 | −15 | +25 |
| CPU 戦勝利 | +10 | — |
| CPU 戦敗北 | −5 | — |
| 引き分け（30 ターン決着なし） | ±0 | ±0 |

- 初期レーティングは 1000、下限は 0。
- レーティングの確定は `POST /battles/ranked` のサーバー処理でのみ行う。
  クライアントから勝敗・デルタを受け取らない（[`docs/09`](09_lightweight_server_architecture.md) 信頼モデル）。
- レーティング差を考慮した Elo 式への移行は未決事項（[`docs/08`](08_mvp_and_roadmap.md)）。
- マッチングはキュー登録済みチームからレーティング差 300 以内の最近接相手を選ぶ。
  相手がいない場合は CPU チームと対戦する。

### ランキング（MVP）

- `GET /ranking` はレーティング降順の上位 20 件（表示名とレーティング）を返す。
- 同率時の並び順は未定義とする（Elo 化と合わせて確定する。[`docs/08`](08_mvp_and_roadmap.md) 未決事項）。
- ページネーション・シーズン別ランキングは将来拡張（Phase 4）。
