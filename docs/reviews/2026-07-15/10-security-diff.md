# ブランチ差分セキュリティレビュー

## 対象

- 比較元: `develop`
- 比較対象: `feature/docs-environments-design` のコミット済み・未コミット差分
- 重点: SQLite 永続化、DB パス解決、Android 開発スクリプト、環境設計ドキュメント

## 結論

リモート攻撃者が HTTP/API 経由で悪用できる Medium 以上の脆弱性は確認されなかった。

## 確認結果

### SQLite 永続化と DB パス

- `PMB_DB_PATH` / `PMB_DATA_DIR` はプロセス環境変数であり、HTTP 入力から到達しない。
- 任意パス指定はデプロイ担当者の信頼境界内で、リモートのパストラバーサル経路ではない。
- 認可ロジックは今回の差分で変更されていない。
- `PMB_ENV` による本番起動ガードは未実装だが、文書上の既知 TODO であり、今回の差分が新たな攻撃経路を開いた証拠はない。
- 共有 Unix ホストでは SQLite ファイルが umask 次第で他ユーザーから読める可能性がある。単一ユーザー開発端末では現実性が低いため正式 finding とはしないが、本番では `0600` 相当への硬化を推奨する。

### Android 開発スクリプト

- 引数は引用されて Gradle に渡され、外部入力を受けるサービスではない。
- 自動ポート検出先は `127.0.0.1:8000` / `8001` の `/docs` に限定される。
- debug の cleartext 通信と release の HTTPS 制約は分離されている。
- 将来 CI から信頼できない入力を渡す設計に変える場合は、`--api-url` の URL 検証が必要になる。

### その他

- 差分内にハードコードされた秘密情報はない。
- 新規 HTTP ハンドラ、課金権限分岐、CSRF/XSS 面の変更はない。
- テストの `store` は引き続き `:memory:` に差し替えられ、認可テストの隔離を壊す具体的経路は確認されなかった。

## 防御的改善

1. 本番公開前に `PMB_ENV=production` の起動時ガードを実装する。
2. 共有ホストまたは本番で SQLite ファイルを所有者のみ読み書き可能にする。
3. 開発スクリプトを信頼できない入力から呼ぶ用途へ拡張する場合、API URL を検証する。
